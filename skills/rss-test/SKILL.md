# 功能测试 Skill

## 触发条件

- code-review 完成且无 BLOCKER 后自动触发
- 用户手动触发："帮我测试""验证功能""测试一下""跑一下测试"

## 前置条件

1. `specs/<feature>/spec.md` 必须存在
2. 代码已编写完成且 code-review 通过
3. 服务可启动

## 执行流程

### Step 1: 读取测试依据

1. 读取 `specs/<feature>/spec.md` — 提取所有接口定义
2. 读取 `specs/<feature>/plan.md` — 确认哪些任务已完成
3. 加载 `REFERENCE/test-strategy.md` — 测试策略规范

### Step 2: 确认服务状态

1. 检查服务是否已运行
2. 如果未运行，提示用户启动服务
3. 等待服务就绪后再执行测试

### Step 3: 编写并执行测试用例

针对 spec.md 中的每个接口，按以下顺序执行：

**3.1 正常流程测试**

对每个接口构造合法请求，验证：
- HTTP 状态码符合预期
- 响应格式正确
- 成功字段为成功值
- 数据字段包含预期的数据结构
- 必要字段存在且类型正确

**3.2 参数验证测试**

对每个接口测试边界条件：
- 缺少必填参数 → 应返回错误
- 参数类型错误 → 应返回错误
- 参数值越界 → 应返回错误
- 空值/nil → 应正确处理

**3.3 权限测试（如适用）**

- 无 Token 访问后端接口 → 应返回认证错误
- Token 过期 → 应返回认证错误
- 无权限 → 应返回授权错误

**3.4 业务逻辑验证**

对照 spec.md 中的业务规则：
- 创建操作后查询确认数据已写入
- 更新操作后查询确认字段已变更
- 删除操作后查询确认状态已变更
- 关联数据是否正确联动

### Step 4: 输出测试报告

格式如下：

```markdown
## 测试报告 — <feature>

**测试时间：** YYYY-MM-DD HH:mm
**服务地址：** http://localhost:port
**接口总数：** N

### 测试结果汇总

| 接口 | 方法 | 路径 | 正常流程 | 参数验证 | 权限 | 业务逻辑 | 结果 |
|------|------|------|---------|---------|------|---------|------|
| xxx | POST | /xxx/edit | ✅ | ✅ | — | ✅ | PASS |
| xxx | GET | /xxx/list | ✅ | ⚠️ | — | ✅ | WARN |

### 失败详情

#### F1: [接口名] [测试类型]
- **请求**：`POST /xxx/edit` body: {...}
- **期望**：code=0, data.id 存在
- **实际**：code=50001, message="创建失败"
- **可能原因**：...

### 需修复项

- [ ] F1: ...
```

### Step 5: 判定结果

- **全部 PASS** → 通过，进入 index-update 步骤
- **存在 FAIL** → 不通过，列出需修复的问题，等待用户确认修复后重新测试
- **存在 WARN** → 警告并且阻塞，必须等用户选择跳过或修复

## 约束

- 测试必须基于 spec.md 中的接口定义，不可凭空猜测
- 测试用例必须覆盖正常流程 + 至少一种异常场景
- 不可修改业务代码（只读），发现问题应报告而非自行修复
- 测试完成后将报告追加到 `specs/<feature>/test-report.md`

## 完成条件与下一步

**测试完成后：**

1. 全部 PASS（或 WARN 已被用户确认跳过） → **触发 index-update skill**
2. 存在 FAIL → **禁止进入下一步**，必须先修复问题后重新测试

**测试通过不等于工作完成。未触发 index-update 视为流程中断。**
