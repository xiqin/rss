# 测试策略规范

## 测试类型

### 1. 单元测试

验证单个函数/方法的正确性。

**适用场景：**
- Service 层业务逻辑
- Model 层查询方法
- 工具函数

**测试模式：**
- 使用项目测试基础设施获取测试 Repository
- 为每个公开方法编写测试
- 覆盖正常流程 + 异常流程

### 2. 集成测试

验证多个组件协同工作的正确性。

**适用场景：**
- API 接口测试
- 数据库 CRUD 操作
- 缓存读写
- 队列消息处理

**测试模式：**
- 启动完整服务
- 使用 HTTP 客户端调用接口
- 验证响应格式和内容

### 3. 端到端测试

验证完整业务流程。

**适用场景：**
- 多步骤业务流程
- 跨模块协作
- 用户完整操作流程

## 测试覆盖维度

### 接口测试清单

对每个接口，测试以下维度：

| 维度 | 测试内容 | 期望结果 |
|------|---------|---------|
| 正常流程 | 合法请求 | 成功响应 |
| 必填参数 | 缺少必填参数 | 参数错误 |
| 参数类型 | 类型错误 | 参数错误 |
| 参数边界 | 越界值 | 参数错误 |
| 权限认证 | 无 Token | 认证错误 |
| 业务规则 | 违反业务规则 | 业务错误 |

### 业务逻辑测试清单

| 场景 | 测试内容 | 验证方式 |
|------|---------|---------|
| 创建 | 数据写入 | 查询确认 |
| 更新 | 字段变更 | 查询确认 |
| 删除 | 状态变更 | 查询确认 |
| 查询 | 条件过滤 | 结果验证 |
| 分页 | 分页逻辑 | 总数和数据验证 |
| 缓存 | 缓存读写 | 命中率验证 |

## 测试数据准备

### 测试数据原则

1. **隔离性**：每个测试使用独立数据，不依赖其他测试
2. **可重复**：测试可多次运行，结果一致
3. **可清理**：测试后清理创建的数据

### 测试数据模式

```
1. 准备（Arrange）：创建测试数据
2. 执行（Act）：调用被测试方法
3. 断言（Assert）：验证结果
```

## 测试报告格式

```markdown
## 测试报告 — <feature>

**测试时间：** YYYY-MM-DD HH:mm
**测试范围：** xxx
**接口总数：** N

### 结果汇总

| 类型 | PASS | WARN | FAIL |
|------|------|------|------|
| 单元测试 | N | N | N |
| 集成测试 | N | N | N |

### 详细结果

#### 接口: POST /xxx/edit
- ✅ 正常流程
- ✅ 参数验证
- ⚠️ 权限测试（未测试）
- ✅ 业务逻辑

### 失败详情

#### F1: POST /xxx/edit 参数验证
- 请求: `{}`
- 期望: code != 0
- 实际: code = 0
- 原因: 缺少参数校验

### 需修复项

- [ ] F1: POST /xxx/edit 缺少参数校验
```

## 测试判定规则

| 结果 | 含义 | 处理 |
|------|------|------|
| PASS | 测试通过 | 继续 |
| WARN | 有未覆盖的测试 | 用户确认后继续 |
| FAIL | 测试失败 | 必须修复后重测 |
