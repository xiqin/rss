# 任务拆分规范

## 拆分原则

### 按分层拆分

严格按以下顺序拆分：

1. **Model 层** → 数据模型、查询方法
2. **Service 层** → 业务逻辑、校验
3. **Controller 层** → HTTP 接口、参数绑定
4. **Router 层** → 路由注册
5. **Config 层** → 配置项、DDL

### 按模块拆分

每个功能点独立为一个任务，标注依赖关系。

## 输出格式

```markdown
# 任务拆分清单

## 总览
- 任务总数：N
- 预计复杂度：简单 X / 中等 Y / 复杂 Z

## Task 1: <功能点>

- **层级**: Model / Service / Controller / Router / Config
- **复杂度**: 简单 / 中等 / 复杂
- **依赖**: 无 / Task X
- **涉及文件**:
  - 创建: `path/to/new/file.go`
  - 修改: `path/to/existing/file.go`

## Task 2: ...
```

## 复杂度判定标准

| 等级 | 标准 | 示例 |
|------|------|------|
| 简单 | 单文件修改，无依赖 | 新增一个查询方法 |
| 中等 | 2-3 个文件，有依赖 | 新增 Service + Controller |
| 复杂 | 多文件，有外部依赖 | 新功能模块，涉及多层 |

## 依赖关系标注

- **无依赖**：可独立开发
- **依赖 Task X**：必须等 Task X 完成后才能开始
- **可并行**：与其他 Task 无依赖关系，可并行开发
