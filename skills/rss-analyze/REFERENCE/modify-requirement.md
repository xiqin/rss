# 修改类需求分析规范

## 前置步骤

在分析修改类需求前，必须先分析现有代码的影响范围：

### 影响分析清单

1. **直接依赖**：哪些文件/函数直接引用了被修改的代码
2. **间接依赖**：通过接口、事件、消息队列等间接关联的模块
3. **数据库影响**：表结构变更是否影响现有查询
4. **API 影响**：接口变更是否影响现有调用方
5. **配置影响**：是否需要新增/修改配置项

## 输出格式

```markdown
# <功能名> 修改需求分析

## 概述
修改目标和范围。

## 影响范围

### 直接影响
| 文件 | 方法 | 影响类型 | 说明 |
|------|------|---------|------|
| path/to/file.go | MethodName | 修改 | 需要扩展逻辑 |

### 间接影响
| 模块 | 接口/事件 | 影响类型 | 说明 |
|------|----------|---------|------|
| xxx-service | API | 兼容性 | 响应格式变更 |

## 修改清单

### 修改点1：XXX
- **文件**：`path/to/file.go`
- **方法**：MethodName
- **现状**：当前实现
- **变更**：修改内容
- **影响**：对其他模块的影响

## 兼容性分析

| 变更项 | 向前兼容 | 向后兼容 | 处理方式 |
|--------|---------|---------|---------|
| API 响应 | ✅ | ✅ | 新增字段 |
| DB 表结构 | ❌ | ✅ | 需要 DDL |

## 回滚方案
- 回滚步骤
- 数据迁移回滚

## 待决议列表
- [ ] 问题1
```

## 约束

- 必须列出所有受影响的文件
- 必须分析兼容性
- 必须提供回滚方案
- 数据库变更必须提供 DDL
