## 审查维度

### 维度 1：架构合规性（BLOCKER）

- Controller 是否包含业务逻辑 → 必须移到 Service
- 是否跨层调用（如 Controller 直接调 Model）
- 新功能是否放在正确的模块/路由分组下
- 依赖是否单向流动

### 维度 2：代码质量（BLOCKER）

- 命名规范：是否遵循项目命名惯例
- 错误处理：所有 error 是否被正确处理，有无静默吞掉
- 日志规范：是否使用项目日志组件，有无 `fmt.Println`
- 函数长度：单个函数是否过长（建议拆分）
- 参数校验：HTTP 接口是否校验了所有必填参数
- 空指针：指针使用前是否判空

### 维度 3：安全风险（BLOCKER）

- SQL 注入：是否使用参数化查询，有无字符串拼接 SQL
- 敏感信息：密码、密钥、Token 是否硬编码
- 权限校验：接口是否经过正确的中间件认证
- 输入校验：用户输入是否经过验证和清洗
- 信息泄露：错误响应是否暴露了内部细节

### 维度 4：性能隐患（WARNING）

- N+1 查询：循环内是否有数据库查询
- 循环内 IO：循环内是否有网络请求或文件操作
- 大事务：事务范围是否过大
- 缓存使用：高频查询是否使用了缓存
- 分页查询：列表查询是否限制了最大返回条数

### 维度 5：规范一致性（SUGGESTION）

- 响应格式：是否使用项目统一响应格式
- 错误码：是否使用项目统一错误码体系
- 国际化：用户可见的错误信息是否走 i18n（如适用）
- 注释：导出函数是否有注释
- 文件组织：文件是否放在正确的目录下

## 执行流程

1. 读取 `specs/<feature>/spec.md`（需求文档）
2. 读取 `.claude/rules/project-structure.md`（工程约束）
3. 逐文件审查，按上述维度检查
4. 输出问题清单，按严重程度分类：
   - **BLOCKER**：必须修复，否则不能合并
   - **WARNING**：强烈建议修复
   - **SUGGESTION**：可选优化
5. 对 BLOCKER 自动修复（如可能）
6. WARNING 和 SUGGESTION 列出，等用户确认

## 完成条件与下一步

**审查完成后：**

1. 如果存在 BLOCKER 且未全部修复 → **禁止进入下一步**，必须先修复所有 BLOCKER
2. 如果无 BLOCKER（或已全部修复） → 继续流水线下一步
