# 代码审查维度规范

## 审查清单

### 维度 1：架构合规性（BLOCKER）

| # | 检查项 | 检查方法 |
|---|--------|---------|
| 1.1 | Controller 是否包含业务逻辑 | 检查是否有 if/for 判断业务规则、数据转换、多步操作 |
| 1.2 | Service 是否直接操作数据库 | 检查是否有直接的数据库操作调用 |
| 1.3 | 是否跨层调用 | Controller 是否直接调 Model，Service 是否直接操作外部存储 |
| 1.4 | 模块分组是否正确 | 功能是否放在正确的模块/目录下 |
| 1.5 | 依赖方向是否正确 | 是否存在循环依赖 |

### 维度 2：代码质量（BLOCKER）

| # | 检查项 | 检查方法 |
|---|--------|---------|
| 2.1 | 是否使用禁止的函数 | 搜索 `fmt.Println`、`log.Print`、`println` |
| 2.2 | SQL 是否参数化 | 搜索字符串拼接 SQL |
| 2.3 | 是否有硬编码密钥/密码 | 搜索可疑字符串：密码、token、secret |
| 2.4 | 是否修改了自动生成文件 | 检查自动生成文件是否有 diff |
| 2.5 | 是否修改了依赖注入生成文件 | 检查 DI 生成文件是否有手动修改 |

### 维度 3：安全风险（BLOCKER）

| # | 检查项 | 检查方法 |
|---|--------|---------|
| 3.1 | SQL 注入 | 参数化查询，禁止字符串拼接 |
| 3.2 | 接口认证 | 新增接口是否有中间件保护 |
| 3.3 | 输入校验 | 是否使用框架校验机制 |
| 3.4 | 信息泄露 | 错误响应是否暴露堆栈、SQL、内部路径 |
| 3.5 | 敏感数据 | 日志是否打印了密码、Token 等敏感信息 |

### 维度 4：性能隐患（WARNING）

| # | 检查项 | 检查方法 |
|---|--------|---------|
| 4.1 | N+1 查询 | for 循环内是否有数据库查询 |
| 4.2 | 循环内 IO | for 循环内是否有 HTTP 请求 |
| 4.3 | 大事务 | 事务范围是否过大 |
| 4.4 | 缺少分页 | 列表查询是否限制了返回条数 |
| 4.5 | 缺少缓存 | 高频查询是否使用了缓存 |

### 维度 5：规范一致性（WARNING）

| # | 检查项 | 检查方法 |
|---|--------|---------|
| 5.1 | 命名规范 | 是否遵循项目命名惯例 |
| 5.2 | 响应格式 | 是否使用项目统一响应格式 |
| 5.3 | 构造函数 | 是否使用标准构造函数模式 |
| 5.4 | 路由注册 | 是否按项目规范注册路由 |
| 5.5 | 注释 | 导出函数/类型是否有注释 |

### 维度 6：代码质量（SUGGESTION）

| # | 检查项 | 检查方法 |
|---|--------|---------|
| 6.1 | 函数长度 | 单个函数是否过长 |
| 6.2 | 注释质量 | 注释是否准确、必要 |
| 6.3 | 魔法数字 | 是否有硬编码数字（应定义为常量） |
| 6.4 | 代码重复 | 是否有大段重复代码可抽取 |
| 6.5 | TODO/FIXME | 是否遗留了未处理的 TODO |

## 审查输出格式

```markdown
# 代码审查报告

## 总览
- 审查范围：xxx
- 问题总数：N
  - BLOCKER：X 个
  - WARNING：X 个
  - SUGGESTION：X 个

## BLOCKER（必须修复）

### B1：[问题标题]
- **文件**：`path/to/file.go:45`
- **维度**：架构合规性
- **问题**：Controller 中包含业务逻辑
- **建议**：将业务逻辑移到 Service 层

## WARNING（强烈建议修复）

### W1：...

## SUGGESTION（可选优化）

### S1：...

## 自动修复
以下 BLOCKER 已自动修复：
- B1：已将业务逻辑从 Controller 移到 Service
```
